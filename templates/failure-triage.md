# 失败分诊决策树（failure-triage.md）

> **用途**：AI / 开发者失败时第一件事——分诊，决定"该不该重试"。
> **何时用**：任何阶段失败立即打开本文件，30 秒决定下一步。
> **设计原理**：A 真问题、B 路径不通、C 脚本时序 —— 把"该不该重试"的糊涂账变清晰决策。

---

## 三类问题核心定义

| 类 | 一句话定义 | 关键判据 | 决策 |
|---|---|---|---|
| **A 真问题** | **底层有 bug，期望结果就是没出现** | 修代码 / 修逻辑 | ⛔ 回「实现」阶段重做 |
| **B 路径不通** | **验证方式错了 / 没有数据 / 不可复现** | 修验证计划 | ⛔ 改 verify_plan，不动代码 |
| **C 脚本时序** | **工具没用对，时机不对，参数错** | 重跑 / 调参数 | ✅ 阶段内重试 ≤ 2 轮 |

> **A 和 B 的根本区别**：
> - A 是**代码有问题**——物存在但坏了
> - B 是**验证方式有问题**——可能根本没找到对的物去测
>
> 把 A 当 B 处理 = 反复重试无效代码；把 B 当 A 处理 = 改对了代码但还在用错的验证方式自我怀疑。

---

## 决策树（顺序问完即得答案）

```
[失败发生了]
  │
  ├─ Q1: 期望的现象/结果出现了吗？
  │     │
  │     ├─ 没出现 ───────────────────→ 【继续 Q2】
  │     │
  │     └─ 出现了但行为不对 ─────────→ A 真问题（回实现阶段）
  │
  ├─ Q2: 我现在验证的对象/路径/账号，是"对的那个"吗？
  │     │
  │     ├─ 不是/不确定 ───────────────→ B 路径不通（修订 verify_plan）
  │     │
  │     └─ 是的 ──────────────────────→ 【继续 Q3】
  │
  ├─ Q3: 工具/脚本/参数/时序对吗？
  │     │
  │     ├─ 不对 ─────────────────────→ C 脚本时序 ✅ 阶段内重试
  │     │
  │     └─ 都对 ─────────────────────→ 【A 真问题】（回实现阶段）
  │
  └─ Q4（如果上轮已经重试过）：本类是 A/B/C 哪个？
        │
        ├─ A：已经回实现阶段修了 → 还有问题 → 升级用户
        ├─ B：已经修过 verify_plan → 还有问题 → 升级用户
        └─ C：阶段内已重试 ≥ 2 轮 → 还有问题 → 升级用户
```

---

## A 真问题（代码 bug）的典型症状

| 症状 | 例子 |
|---|---|
| 期望数据未生成 | "我调用 API 后数据库没记录" |
| 输出值错 | "算的订单总额 = ¥0，但应该是 ¥199" |
| 期望 UI 没出现 | "登录后没看到用户头像" |
| 期望事件未触发 | "按钮点击后回调没跑" |
| 编译/测试报错 | "go build 退出码 1，类型不匹配" |

**对应动作**：
- ⛔ 立刻停止当前验证
- ⛔ 回到"实现"阶段，修复代码
- ⛔ 修复后再跑一遍验证（**不允许在验证阶段继续硬试**）

---

## B 路径不通（验证方式错）的典型症状

| 症状 | 例子 |
|---|---|
| 被登录页拦截 | "试图打开主页但跳到登录" |
| 没有测试数据 | "数据库空，跑 100 遍也是空" |
| 真机才有现象，模拟器不出现 | "iOS 真机有 bug，模拟器跑不出来" |
| 后台/异步未到时间 | "消息没推，但可能还没到 30 秒延迟" |
| 缓存命中老数据 | "看上去对的，是浏览器缓存没清" |

**对应动作**：
- ⛔ 改 `verify_plan.md`，重写验证路径
- ❌ **不动代码**——这不是代码的锅
- ✅ 加辅助步骤：登录、灌数据、清缓存

---

## C 脚本时序（执行细节问题）的典型症状

| 症状 | 例子 |
|---|---|
| 元素没渲染就点击 | "脚本点按钮，元素还在 loading" |
| 坐标算错/视窗不对 | "点击位置偏了 30 像素" |
| sleep 不够 | "等 1 秒不够，等 5 秒就 OK" |
| 端口/进程冲突 | "8888 端口已占用" |
| 临时网络失败 | "curl: (7) Failed to connect，重试成功" |

**对应动作**：
- ✅ 阶段内重试 ≤ **2 轮**（不是无限重试）
- ✅ 第 1 次重试：调 sleep、加 wait、上一步前置状态检查
- ✅ 第 2 次重试：改用更稳的方式（显式等元素 / 改端口）
- ⛔ 第 3 次还有问题 → 升 A 或升用户，**不是继续硬试**

## 反直觉坑（最容易掉进去的）

### 坑 1：把 B 当 A，看不见"我没在测对的东西"

> 失败现象：用户头像没显示。
> 误判：A（前端 bug）。
> 实际：B（账号权限不够，看到的是占位图）。
>
> **识别**：在 verify_plan.md 写下"我打算验证的路径"——如果验证路径里漏了"用对账号登录 / 清缓存"，那大概率是 B。

### 坑 2：把 A 当 C，死循环重试

> 失败现象：API 返回 500。
> 误判：C（脚本时序，重试就好）。
> 实际：A（后端代码 panic，从来不会好）。
>
> **识别**：重试 1-2 次后看**规律**——如果每次失败现象一模一样，毫无变化，A 的概率飙升。

### 坑 3：把 C 当 A，回实现阶段改无关代码

> 失败现象：模拟器点不中按钮。
> 误判：A（前端样式错）。
> 实际：C（坐标 / 元素 selector 写错）。
>
> **识别**：在浏览器/IDE 里手动复现一次——如果手动能成功，就是 C。

---

## 升级用户的标准（什么时候该停下问）

满足以下**任一**条件，立刻停下来问用户，不要硬试：

1. 同一类问题已经回实现阶段改过 2 次，还没好
2. 同一脚本重试 ≥ 2 轮还没好
3. 涉及外部 API / 凭证 / 计费 / 用户数据
4. 涉及删文件 / push / 发邮件 / 任何不可逆操作
5. 失败现象完全没见过，无从判断 A/B/C

---

## 一句话总结

> **失败分诊的本质是回答两个问题**：
> 1. 我在测的东西**对不对**？不对 → B
> 2. 我测的方式**对不对**？不对 → C
> 3. 都没问题但还是失败 → A
>
> **永远先验证"我在测对的东西"，再怀疑代码。**
