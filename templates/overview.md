# 项目总览模板（overview.md）

> **用途**：项目级 L1 文档，**≤ 5KB**，给 AI 和新人都能 5 分钟看懂项目全貌。
> **何时写**：项目雏形成立时（哪怕只有 2 个文件，也值得写）。
> **抄走方式**：复制本文件 → 改名 `PROJECT_WIKI/overview.md` → 填空。

---

## 为什么先写这个

不写 = 每次新会话 AI 都从零开始理解你的项目，重复踩同样的坑；
写了 = 下次 AI 接力读 5 分钟就能干活，新人入职也靠这一份入门。

---

## 模板（blank，照抄改）

```markdown
# <项目名> 项目总览

> 创建日期：YYYY-MM-DD
> 最近更新：YYYY-MM-DD（commit hash 可选）
> 维护者：<名字 / 联系方式>

## 项目一句话

<不超过 30 字，说清"这个项目解决谁的什么问题">

## 当前状态

- 🎯 **正在做**：<本周在做的那一件事>
- ⏭️ **下一步**：<明确的下一步动作>
- 🚫 **暂停**：<做了一半暂停的事，写明为什么暂停>

## 模块地图

| 模块 | 路径 | 一句话职责 |
|---|---|---|
| <模块 1> | `src/<path>/` | <这一块干什么> |
| <模块 2> | `src/<path>/` | <同上> |
| <模块 3> | `<path>` | <同上> |

> **只看这张表，AI 应该能判断"用户的需求该改哪个模块"。**

## 关键不变量（Invariants）

> 这些是"项目里绝对不能动"的东西。新人改之前必须三思。

1. <不变量 1：例 "API 路径前缀 `/v1/` 不能换，会破坏旧客户端">
2. <不变量 2>
3. <不变量 3>

## 技术栈与依赖

- **语言**：<Go / Python / TypeScript / ...>
- **核心依赖**：<3-5 个最关键的库，不要列小工具>
- **运行要求**：<Python 3.11+ / Go 1.23+ / Node 20+>
- **部署方式**：<本地 / Docker / K8s / Serverless>

## 常见任务入口

> 新人 / AI 第一周会做的事，每条一句话+文件路径。

- "加一个新功能" → 改 `<file>` 中的 `<function>`，**先看后写，模仿已有约定**
- "修 bug" → 先看 `timeline.txt` 找最近的 commit 上下文
- "改一处配置" → 改 `<config file>`，跑 `<验证命令>` 看退出码

## 失败处理

- 编译失败：自修 ≤ 3 轮，超过立即停下回退
- 端到端验证失败：A 真问题回实现、B 路径不通修验证计划、C 脚本时序重试 ≤ 2 轮
- AI 提议改动但不写测试 → 拒收

## 链接

- 详细 spec：`<spec 路径>`
- 任务台账：`<tasks 路径>`
- 时间线：`<timeline 路径>`
```

## 填好示例（GitHub issue 爬虫小工具）

```markdown
# gh-issue-crawler 项目总览

> 创建日期：2026-07-15
> 最近更新：2026-07-21
> 维护者：<name>

## 项目一句话

从指定 GitHub 仓库拉 issues + comments，按"讨论热度"排序，输出 markdown 周报。

## 当前状态

- 🎯 **正在做**：支持多仓库切换（轮子里的硬编码仓库路径 → 配置项）
- ⏭️ **下一步**：写单测覆盖排序逻辑
- 🚫 **暂停**：飞书机器人推送（API 凭证没到位，等配置）

## 模块地图

| 模块 | 路径 | 一句话职责 |
|---|---|---|
| fetcher | `internal/fetcher/` | 调 GitHub API 拉数据，含分页和限流 |
| ranker | `internal/ranker/` | 按热度给 issue 排序（comment 数 + 时间衰减） |
| writer | `internal/writer/` | 输出 markdown，模板在 `templates/report.tmpl` |
| main | `cmd/gh-issue-crawler/main.go` | 编排 + 配置加载 + CLI 入口 |

## 关键不变量（Invariants）

1. **GitHub API 限流**：60 次/小时（无 token），6000 次/小时（有 token）。代码里不能硬编码超过这个上限。
2. **排序逻辑确定性**：相同输入必须产出相同排序，避免 AI 输出"看起来差不多"的结果。
3. **markdown 模板不动源码**：模板文件改 `templates/`，不在 Go 代码里硬编码。

## 技术栈与依赖

- **语言**：Go 1.23+
- **核心依赖**：`github.com/google/go-github/v62`、`gopkg.in/yaml.v3`
- **运行要求**：网络可达 `api.github.com`
- **部署方式**：本地二进制 + cron

## 常见任务入口

- "加一个新排序维度" → 改 `internal/ranker/ranker.go` 的 `Score()` 函数，先看后写
- "输出格式改一下" → 改 `templates/report.tmpl`，跑 `go test ./...`
- "接入新仓库" → 改 `configs/repos.yaml`，重启进程

## 失败处理

- GitHub 限流触发 → 等 1 小时或加 token
- 排序结果不稳定 → 加单元测试固化预期
- markdown 输出缺字段 → 看 `templates/report.tmpl` 和 `writer.Render()` 的字段映射

## 链接

- 详细 spec：`docs/spec.md`
- 任务台账：`tasks/`
- 时间线：`runs/timeline.txt`
```

---

## 常见错误

| 错 | 正 |
|---|---|
| 写成项目宣传介绍 | 写成"地图 + 不变量 + 任务入口"，不写品牌故事 |
| > 5KB | 内容多了就拆出 `docs/modules/` 子文件，overview 保持精简 |
| 没"当前状态"小节 | 这一节是核心：让任何接手的人 5 秒知道"现在在干啥" |
| 不变量写得太抽象 | "用户数据安全"（错）→ "API 强制走 HTTPS，证书由 <service> 管理"（对） |
