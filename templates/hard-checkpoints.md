# 硬关卡清单（hard-checkpoints.md）

> **用途**：把"AI 跑得太快忘了让人点头"前置成"必须等用户拍板才执行"的清单。
> **何时用**：项目立项时定义好 HK 列表 + 模板化承诺语，写入 red_lines.yaml 联动。
> **设计原则**：AI 跑得飞快，但**任何不可逆动作都先让人点头**。

---

## 核心思想

AI Agent 在长任务里跑 10-30 分钟，期间会做大量决策。如果每个决策都问用户，效率归零；如果所有决策都不问，出错时救不回来。

**HK（Hard Checkpoint，硬关卡）的解法是中间路线**：

- 可逆决策（写代码、改文档、调参数）→ AI 自主
- 不可逆决策（发邮件、git push、删文件、外部 API 写、改线上配置）→ **HK 触发，必须停下等人点头**

> **信任但不放任**。

---

## 通用 HK 清单（v0.1）

> 直接抄走，按项目扩展。每个 HK 配一个 trigger / response template / 升级路径。

### HK-0 · 现场快报（接力场景）

| 项 | 说明 |
|---|---|
| **触发时机** | 新会话接力入口进入后第一时间（接手上一个 AI 的工作） |
| **目的** | 让新会话的 AI 报告"现在项目长啥样 / 卡在哪 / 下一步是啥" |
| **等用户回** | "确认 / 改 N" |

承诺语模板：

```
🙋 HK-0 现场快报

📍 当前进度：<一句话>
📍 卡在哪：<一句话>
📍 下一步建议：<一句话>
📍 涉及文件：<路径列表>

请回：「确认」/「改 N」/「跑 X 命令先看看」
```

---

### HK-1 · PENDING 条目（拆解完成后）

| 项 | 说明 |
|---|---|
| **触发时机** | 任务"翻译/拆解"完成后，正式进入实现前 |
| **目的** | 让用户对"接下来要按这个顺序干这几件事"做最后确认 |
| **等用户回** | "确认 / 改 xxx / 加一项" |

承诺语模板：

```
🤝 HK-1 PENDING

📋 接下来要做（按顺序）：

1. <子任务 1> — 改 <文件> — 预计 <时间>
2. <子任务 2> — 改 <文件> — 预计 <时间>
3. <子任务 3> — 改 <文件> — 预计 <时间>
4. ...
5. 最终提交 / 发布

请回：「确认」/「改 xxx」/「加一项」
```

---

### HK-2 · 沉淀 OK（文档落盘前）

| 项 | 说明 |
|---|---|
| **触发时机** | 任何涉及"项目级文档落盘"前（spec.md 改完 / overview 改完 / 演进事件表更新） |
| **目的** | 文档是给"未来的自己"看的，落盘前必须人眼过一遍 |
| **等用户回** | "沉淀 ok / 通过 / 改 xxx" |

承诺语模板：

```
📝 HK-2 沉淀

📄 文档路径：<file>
🔄 改动摘要：
  - 加：<new chapter>
  - 改：<modified section>
  - 删：<removed>
📋 改动行数：+N / -N

[可选预览：前 30 行]

请回：「沉淀 ok」/「改 xxx」/「撤回」
```

---

### HK-3 · 提交 / 发布文案（外部动作前）

| 项 | 说明 |
|---|---|
| **触发时机** | `git commit` / git push / 外部 API 写 / 发邮件 / 发推 / 改线上配置 之前 |
| **目的** | 这是不可逆动作！文案一旦发出改不回来 |
| **等用户回** | "提交 / go / 改文案" |

承诺语模板：

```
🚀 HK-3 提交 / 发布

📌 动作：<git commit / push / send / post / config-update>
📝 目标：<commit hash 待生成 / 收件人 / 平台>
📋 内容预览：

---
<commit message draft / email body / post content>
---

⚠️ 影响范围：
  - 触及文件：<list>
  - 不可逆：<是 / 否>
  - 公开范围：<private / team / public>

请回：「提交」/「go」/「改文案」/「暂停」
```

---

### HK-4 · 长跑命令启动（>30s / 计费 / 资源密集）

| 项 | 说明 |
|---|---|
| **触发时机** | 任何预计 > 30 秒的命令，或调用计费 API，或重资源操作 |
| **目的** | 长跑命令一旦启动停不下来，先确认是否值得跑 |
| **等用户回** | "跑 / 不跑 / 改超时阈值" |

承诺语模板：

```
⏱️ HK-4 长跑命令

🛠️ 命令：<full command>
⏱️ 预计耗时：<est time>
💰 是否计费：<yes/no, 预估费用>
🛟 失败时回滚：<how>
📋 落盘证据：<what file to check>

请回：「跑」/「不跑」/「先试 30s 看看」/「调超时为 X」
```

---

### HK-5 · 失败回退（已经从 A 阶段修回到上游）

| 项 | 说明 |
|---|---|
| **触发时机** | 失败分诊判定为 A（代码 bug），准备回退到「实现」阶段时 |
| **目的** | 让用户知道"我们要回退重做某件事"，避免反复横跳 |
| **等用户回** | "回退 / 改为 B 分诊 / 直接放弃这子任务" |

承诺语模板：

```
⏪ HK-5 回退决定

🔍 分诊结果：A 真问题（代码层 bug）
📍 当前位置：<stage N>
📍 准备回退到：<stage M>
⏱️ 预计重做耗时：<est time>

回退方案（建议）：
  1. 回 stage M 重新产出 X
  2. 重跑 stage M+1 → stage N
  3. 重新验证

请回：「回退」/「改方案」/「放弃这子任务」
```

## 触发哪些 / 不触发哪些 · 项目级配置

```yaml
# red_lines.yaml 的扩展段
hard_checkpoints:
  enabled:
    - HK-0   # 现场快报
    - HK-1   # PENDING 条目
    - HK-2   # 沉淀 OK
    - HK-3   # 提交/发布
    - HK-4   # 长跑命令
    - HK-5   # 失败回退
  disabled: []
  custom:
    - id: HK-PROJ-1
      trigger: "本项目特有的 X 不可逆动作"
      template: |
        <自定义模板>
```

**配置原则**：

- ✅ 项目里的"对外不可逆动作"必须有 HK
- ❌ 项目里"纯本地可逆操作"不需要 HK（写代码、跑测试、改文档——除非改的是项目级文档）
- ⚖️ 平衡：HK 太多 = 体验差；HK 太少 = 失控

---

## 一句话原则

> **可逆的事让 AI 跑，不可逆的事让人拍。**
>
> 信任 AI 写代码的细节判断，但不信任 AI 自己决定要不要 push / 发邮件 / 删文件 / 改线上。

---

## 反模式

| 错 | 正 |
|---|---|
| HK 模板用一段散文 | 模板必须**填空式** + emoji 分块，方便扫读 |
| HK 触发后 AI 自己决定"应该不算不可逆，跑吧" | 触发即停 + 模板化承诺语，AI 不允许自行判断"够不够不可逆" |
| HK 全套上 = 7 个 | 默认 HK-0 ~ HK-5 即可，按需扩 |
| HK 完成后不写 trigger_log | 每次 HK 触发要写时间戳 + 用户响应，下次回顾哪些 HK 触发最频繁 → 反思流程 |
