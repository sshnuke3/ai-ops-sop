# AI 长任务协作 SOP（通用版）

> **适用对象**：用 AI Agent / LLM 跑「3 步以上长任务」的个人开发者和 5 人以下小团队。
> **不适用**：纯聊天问答、一次性脚本、C 端情感类 AI 应用。
> **目标读者**：在用 Claude / GPT / 国产大模型做事情，但总觉得"AI 时灵时不灵、超长任务跑一半卡了、上下文撑爆"的人。

---

## 0️⃣ 30 分钟抄走（最小启动）

如果你是第一次接触本 SOP，**先做完这 4 件事**就能用上 80% 的价值：

### 5 分钟：读这一节

AI 长任务最大的 3 个坑：
1. **偷懒跳步骤** → 让 AI "凭印象" 完成，而不是校验产物
2. **上下文爆炸** → 让 AI 把用户原话读 N 遍，每次重读都跑飞
3. **不可逆动作失控** → AI 自己决定 push / 发邮件 / 删文件，本应让人点头

本 SOP 的解药 = 上面 3 坑各对应一套机制：**流水线 / 产物 / 硬关卡**。

### 15 分钟：抄走 3 个模板

去 `templates/` 目录：
1. **`task-output-spec.md`** —— 给你下一个长任务写阶段清单
2. **`overview.md`** —— 给你项目写一份 L1 总览（≤ 5KB）
3. **`failure-triage.md`** —— 下次任务失败时 30 秒分诊

### 10 分钟：跑一遍最小流程

- 找一个 3+ 步的小任务
- 复制 `task-output-spec.md` 模板，写 3 阶段 + 退出标准
- 跑任务时每次失败打开 `failure-triage.md` 决策
- 任务完成 → 用 `overview.md` 把项目状态更新一下

> **做完这 4 件事，你就领先 80% 的 AI 用户了。**
> 接下来按需学 `red_lines.yaml`（项目级红线）、`tech-spec.md`（项目级规约）、`hard-checkpoints.md`（硬关卡）。

## §1 流水线骨架：把任何长任务拆成可校验阶段

### 核心思想

把"任务"拆成**严格顺序的 N 个阶段**。每阶段三件套：

| 项 | 要求 |
|---|---|
| **输入** | 上一阶段的产物文件 / JSON / 报告（不是用户原话） |
| **产出** | 明确的文件落盘 + 摘要 |
| **退出标准** | 机器可校验（退出码 0 / 文件存在 / 日志命中 / 截图达标） |

> **关键约束**：后一阶段的输入 = 前一阶段的产出。**不允许回到用户原话**。
> 这一条是 90% AI 长任务失败的根因——AI 偷懒、跳步骤、或重新读一遍原话再次理解。

### 8 阶段参考模板

不同任务重排阶段数，但骨架结构不变：

| # | 阶段 | 关键产出 | 退出标准 |
|---|---|---|---|
| 1 | **设计 / 规划** | 候选清单 + 截图/草图 | 筛选后剩 N 个备选 |
| 2 | **拆解** | 子任务表格 + 任务文件（JSON/YAML） | 每个子任务归宿明确 |
| 3 | **定位** | 文件 + 行号 + 调用链 | 收敛到 < 30K token |
| 4 | **实现** | 代码 / 内容改动 | "先看后写"无新模式 |
| 5 | **验证** | 编译报告 / 测试报告 | 退出码 0，自修复 ≤ 3 轮 |
| 6 | **端到端验证** | 截图 + 日志 | A/B/C 三类诊断分流 |
| 7 | **沉淀** | 文档（带版本号） | 单一事实源落盘 |
| 8 | **提交 / 发布** | git commit / API 调用 | 三段式提交 + AI 署名 |

> **不要照搬阶段数**——根据你的任务重排。
> 关键是：**输入必须是上一阶段产物**，不回到原始 PRD。

### 为什么"输入 = 上一阶段产出"这么重要

反面案例：用户在阶段 1 说了"做一个排版好看的知乎专栏"。
- ❌ 错误流：用户原话 → 阶段 4 实现（AI 从头理解一遍）→ 阶段 5 验证（AI 第二次理解原始需求）→ 阶段 6 端到端验证（AI 第三次理解）……上下文爆炸。
- ✅ 正确流：用户原话 → 阶段 1 产出 `brief.md` → 阶段 2 读 `brief.md` 产出 `tasks.json` → 阶段 3 读 `tasks.json` 产出 `design.png` → ……每阶段只读自己关心的上游产物。

## §2 三层知识地图：让 AI 拥有"项目记忆"

### 金字塔（按需加载）

| 级别 | 文件 | 粒度 | 加载时机 | 体积上限 |
|---|---|---|---|---|
| **L1 总览** | `PROJECT_WIKI/overview.md` | 模块名 + 一句话职责 | 默认 | < 5KB |
| **L2 模块** | `PROJECT_WIKI/<module>.md` | 每个关键文件 + 功能说明 | 命中模块后按需加载 | 单文档 |
| **L3 语义桥** | `wiki/figma_token_mapping.md` 或类似 | 设计/产品术语 → 工程 API | 实现阶段强制加载 | 单文件 |

> **L3 是最被低估却最有价值的一层**——抹平"业务侧"和"工程侧"之间的语义鸿沟。
> 例如：「设计稿说"卡片式列表"」← 这句话对工程毫无信息量，但 `figma_token_mapping.md` 写明：卡片 = `CardView` 组件 + radius=12 + shadow=`shadow_md`。AI 看到这个映射就能一次写对。

### 漂移检测机制

wiki 不是写完就完事，要防漂移：

1. **SHA 基线缓存**（`.review_cache.json`）—— 记录每个 wiki 文件上次审阅时的代码 SHA
2. **三色分诊清单**：每次 commit 后跑一次
   - 🟢 新增文件 → 写新模块文档
   - 🔴 删除文件 → 删对应模块文档
   - 🟡 大改 → 更新对应段落
3. **pre-commit hook 阻断**——stale 信号没清，commit 退出码 1

> 没有这一步，wiki 半年后会跟代码完全脱节，反而成为误导。

## §3 五步定位法：把"大海捞针"压成 < 30K token

| 步 | 给 LLM 的输入 | 输出 | 用 LLM 还是脚本 |
|---|---|---|---|
| 1 | 项目概述（~2K）+ 用户原话 | "这个目标可能对应 N 种技术解读" | LLM |
| 2 | 目录树 + 解读结果 | 2-3 个候选文件路径 | LLM |
| 3 | （不进 LLM） | `rg` / `grep` 跑一遍，命中函数声明 | **脚本** |
| 4 | 单个文件相关片段（~10K） | 完整调用链 | LLM |
| 5 | 函数实现（~5K） | 最终改动点 + 理由 | LLM |

> **真正的诀窍**：
> - 前 2 步**只看目录树**，不读源码
> - 第 3 步**让脚本 grep**，不让 LLM 模拟 grep
> - 第 4 步才真正开始读代码
>
> Token 消耗从"全文 ~10M+" 降到"目标区域 ~30K"——**300× 压缩**。

### 脚本兜底原则

> **LLM 负责"读、判断、决策"，精确数值/幂等执行/批量操作全下沉到脚本。**

反例：让 LLM "找出所有函数中以 `_handler` 结尾的"，100% 出错——LLM 数不对、漏行、虚构。
正例：让 LLM 决定"找什么模式"，脚本 `rg "_handler$" --type go -l` 一行解决。

## §4 红线系统：把"AI 翻车"前置成硬关卡

### 单一真源：YAML 文件

```yaml
# red_lines.yaml 骨架
red_lines:
  - id: RL-15
    level: critical   # critical = 启动加载，standard = 按阶段加载
    title: 编译必须退出码 0
    trigger: 任何"实现完成"声明前
    action: 自修复，硬上限 3 轮
    on_violation: ⛔ 停下，按红线模板汇报
```

### 两层分类

| 级别 | 加载时机 | 例子 |
|---|---|---|
| 🔴 **Critical** | Agent 启动时全量加载 | 编译退出码、UI 视觉对齐、commit 信息完整性 |
| 🟡 **Standard** | 各阶段入口按需加载 | 命名规范、单文件不超过 N 行、单元测试覆盖率 |

### 触发即停 + 模板化报告

任何红线触发，AI 必须停下，按下面这个模板汇报（不要让它自由发挥地解释）：

```
⛔ 触发红线 RL-XX：<标题>
当前情形：<具体说明>
建议处理：<回退步骤 / 需要用户确认>
```

> **设计精髓**：把"AI 偷偷做了不该做的事"变成"AI 主动告诉你它撞上红线了"。
> **可观测性远比聪明更重要。**

详见：`templates/red_lines.yaml`（含 9 条标准红线 + load_policy + 触发记录 schema）

## §5 验证双闸门：从"代码改动"到"敢说做完了"

### 闸门一：编译 / 测试验证

- **退出码 0 = 唯一判据**（AI 自我评估 "我觉得做完了" 不算）
- **A/B 分类**：

| 类 | 现象 | 处理 |
|---|---|---|
| **A 可自修复** | 缺分号、标识符未声明、类型不匹配 | ✅ AI 自修 |
| **B 需人介入** | 编译配置错、第三方依赖错、签名错 | ⛔ 立即停下，绝不硬试 |

- 自修复硬上限 **3 轮**——超过必须停下来报告，不要磨到天荒地老。

### 闸门二：端到端验证（模拟器 / 浏览器 / 真机）

- **第 1 步：路径推导**——从 `git diff` 反推验证路径（写到 `verify_plan.md`）
- **第 2 步：路径预扫描**——3 类桥梁法：
  - 通知 / 事件 / Delegate
  - 跨模块通信
  - 数据流传递
- **第 3 步：执行 + A/B/C 三类诊断**

| 类 | 现象 | 处理 |
|---|---|---|
| **A 真问题** | 代码 bug，期望结果没出现 | ⛔ 回「实现」阶段 |
| **B 路径不通** | 被登录页拦截 / 没数据 / 真机才能复现 | ⛔ 修订 verify_plan |
| **C 脚本/时序** | 元素未渲染就点 / 坐标算错 | ✅ 阶段内重试 ≤ 2 轮 |

> **A/B/C 分类的设计精髓**：把"该不该重试"这个糊涂账变成清晰决策。
> AI 不允许在 A/B 类问题上反复硬试；C 类最多 2 轮重试不过 → 升级用户。

详见：`templates/failure-triage.md`（决策树 + 反直觉坑）

## §6 跨会话知识传承：三件套（不同时间尺度）

| 文件 | 尺度 | 内容 |
|---|---|---|
| **`docs/spec.md`** 或 `PROJECT_WIKI/spec.md` | **永久** | 功能边界、模块地图、不变式、Bug/演进历史 |
| **`tasks/`** 或 `subtasks.json` | **跨会话** | 每个子任务的状态、当前阶段、关联 commit |
| **`runs/timeline.txt`** | **会话内** | 启动 / 人工修正 / 提交 三类事件流水 |

### 文档章节结构

- **§0 AI 自检清单** —— 给下次会话的 AI 当"入场扫描"
- **§1 功能边界** —— 哪些做、哪些不做（防越界）
- **§2 模块地图** —— 文件 + 关键方法 + 调用链
- **§3 不变式** —— 不能动的命名、文件清单、协议边界
- **§4 演进事件** —— 按时间线排列的 BUG-N / ITER-N / REV-N
- **§5 产物清单** —— 每次 commit 改了什么
- **§6 版本号** —— v1.0 → v1.1 → ... → v2.0 (baseline 合并)

> **新会话的 AI 按 §0 → §1 → §2 → §3 顺序读完，就能无缝接力。**
> 不写这些 = 每次都从零开始，每代 AI 都重新踩一遍一样的坑。

详见：`templates/overview.md`（≤5KB 总览）+ `templates/tech-spec.md`（完整 spec）

## §7 硬关卡 HK：信任但不放任

### 通用清单（v0.1 6 条）

| HK | 触发时机 | 用户回什么 |
|---|---|---|
| **HK-0** 现场快报 | 新会话接力进入后 | "确认 / 改 N" |
| **HK-1** PENDING 条目 | 拆解完成后 | "确认 / 改 xxx" |
| **HK-2** 沉淀 OK | 文档落盘前 | "沉淀 ok / 通过 / 撤回" |
| **HK-3** 提交 / 发布文案 | git commit / 发邮件 / push / 任何不可逆动作 | "提交 / go / 改文案" |
| **HK-4** 长跑命令启动 | > 30 秒 / 计费 / 重资源 | "跑 / 不跑 / 调超时" |
| **HK-5** 失败回退 | A 类失败准备回退到「实现」时 | "回退 / 改方案 / 放弃" |

### 设计原则

- **可逆的事让 AI 自主**：写代码、改文档、调参数
- **不可逆的事让人拍**：发邮件、git push、删文件、外部 API 写、改线上配置
- **触发即停** + 模板化承诺语，不允许 AI 自行判断"够不够不可逆"

详见：`templates/hard-checkpoints.md`（每条 HK 含完整承诺语模板）

## §8 5 条普适原则（直接抄）

任何带 AI 的长任务系统，都可以拿这 5 条原则当设计检查表：

| 原则 | 一句话总结 |
|---|---|
| **流水线化** | 把任务拆成 N 个阶段，每阶段输入/输出/退出标准都机器可校验 |
| **脚本兜底** | LLM 负责"读、判断、决策"，精确执行/批量操作下沉到脚本 |
| **红线前置** | 把"AI 绝对不能做的事"写成 YAML + 分层加载，触发即停、模板化报告 |
| **落盘判定** | 任何长跑命令的成功证据都是"文件存在 + 退出码 0"，不依赖 stdout |
| **沉淀闭环** | 每次任务产出一份 git-tracked 的总览文档，让"知识"和"产物"等量齐观 |

## §9 反直觉洞察

1. **"AI 友好"和"新人友好"在这里完全统一了**——同一份 `overview.md` 给 AI 看懂也给新人看懂，不冲突。
2. **真正的提效不在"AI 写代码"，而在"AI 不再需要人来当翻译"**——开发者从"任务翻译机"解放，转为"AI 的产品经理"。
3. **references/ 比 tools/ 体量更大**——绝大部分能力来自被显式编写的规则、知识、模板，而不是"指望模型聪明"。
4. **上下文预算的核心心法**：不要让 AI"重新读一遍原话"——把每段原话在第一次落地时**摘要成结构化产物**，后面只读产物。
5. **失败处理的 3 类分诊（A/B/C）普适**：编译/测试失败、UI/端到端失败、长跑脚本假死——任何"该不该让 AI 重试"的场景都可以套这套。

## §10 你今天就能做的 3 件事（最小启动）

不需要先配齐所有东西。从这 3 件开始：

### 1. 给你的下一个长任务写"产物清单"

打开 `templates/task-output-spec.md`，复制空白模板。哪怕只有 3 个阶段，也明确写出：
- 每个阶段的输入文件 / 输出文件
- 每个阶段的退出标准（退出码？文件存在？日志命中？）

> 这一条做完你就赢了一半。后面的 §2 §3 §5 都依赖这个。

### 2. 给你的项目写一份 `overview.md`

打开 `templates/overview.md`，填 ≤ 5KB，三件事：
- 模块名 + 一句话职责（一行一个）
- 关键不变量
- 当前在做什么、卡在哪

> 不需要 wiki 全套，先有 overview，下游 wiki 自然跟上。

### 3. 失败时分诊一次

下一次任务失败时，**先停一秒**，打开 `templates/failure-triage.md`：是 A 真问题（bug）、B 路径不通（验证方式错）、还是 C 脚本时序？分类决定处理：A 回实现、B 修订验证计划、C 重试 ≤ 2 轮。

> 把"该不该让 AI 再试一次"这个糊涂账变成清晰决策。

## §11 适用范围和边界

### 适合

- ✅ 3+ 步骤的代码任务
- ✅ 多会话、长周期的内容生产
- ✅ 数据处理流水线
- ✅ 跨工具链的复合任务（"先 A 工具，再 B 工具，最后 C 工具"）
- ✅ 个人 / 5 人以下小团队的工程协作

### 不适合

- ❌ 单次问答（直接问答就够，不需要流水线）
- ❌ C 端情感类 AI 应用（个性化产品，规律完全不同）
- ❌ 学术研究类（需要严谨方法论，不是工程纪律）
- ❌ 大团队（30+ 人）—— 这套是为"个人 / 极小团队"准备的，大团队需要更重的流程

### 一个诚实的警告

> **流水线越精细，"AI 提效"越依赖"工程纪律"。**
>
> 如果你或者团队的工程纪律不够，这套东西反而会变成负担——
> 你得先有产物清单、得有 wiki、得有红线 yaml、得有基础脚本、得会写退出标准。
>
> **反过来说**：如果你已经能做到这些基础，AI 提效会从"快一点"变成"以前做不到的事现在能做到了"。
