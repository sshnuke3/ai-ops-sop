# gh-issue-summarizer 时间线样本
#
# 这是 runs/timeline.txt 在示例项目里的填法示范。
# 真实项目里这一文件由 run 主流程每次追加，AI / 维护者不再手写。
# 但**第一次**写它时，本文件里所有的事件都需要被某次 commit 触发"真实发生过"。
#
# 时间线 = SOP §6 三件套之"会话内时间线"
# 三类事件：
#   start        新一次运行的起点
#   human-correction  人工干预（含报错、要求改文案、改字段等）
#   commit       不可逆动作完成（含 git commit / Notion 写入 / LLM 调用）

# === 时间线字段规约 ===
# 事件格式：<时间> <类型> <一句话描述> [commit-hash]
# 时间粒度：HH:MM:SS（不必精确到毫秒）
# 倒序：最新在最上面（最新事件先让人看到）

# === 维护说明 ===
# 1. **每次 commit 后**追加一行（CI hook 自动加，或人工加）
# 2. human-correction 永远写明"维护者看到了什么、决定改什么"——便于半年后回溯决策
# 3. start 是**任务级别**的起点，不是会话级别——同一会话多个任务会写多行
# 4. 倒序排列（最新在最上）——vim/grep 时 "?" 反向搜最近事件更省时间

# === 本 timeline.txt 示例特意展示的 3 件事 ===
# 1. 三类事件齐备（start / human-correction / commit）—— SOP §6 要求
# 2. human-correction 行都写明"看到 + 改什么"——半年后回看能复现当时思路
# 3. commit 行带 commit-hash 短码——便于 `git log --grep` 反查
